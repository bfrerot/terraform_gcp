### Role

We distinguish 3 families of roles:

==> Simple user role
rol-<profile>

profile = [ developer, aidata, poweruser, audit ]

example:
rol-developer
rol-audit

==> Global user role
rol-<profile>-admins

profile           = [ organization, billing, network, monitoring, security, audit, backup ]

example:
rol-billing-admins

==> Resource role
rol-<project label>-<env>-<label*>

env = [dev, uat, pre, prd]

label (optional): free value that helps to distinguish between roles

example:
rol-ecommerce-noprd-dataexporter
rol-ecommerce-noprd-gkeadmin


Role Vocabulary:

- Default Role for Organizational Profiles : Roles (defined at the organization level) for Organizational Groups (GCP LZ admins, GCP Network Admins, GCP Security admins...).
If an organizational group requires additional permissions for a specific project, do not modify the default role. Instead, use a Project-Specific Role for Organizational Profiles.

- Default Role for Business Profiles : Roles (defined at the organization level) for Business Groups (GCP Developers, GCP Aidata, GCP Audit, GCP PowerUser, GCP DataEngineer coming soon...). These roles are BOUND to Business Groups and scoped only to their respective projects. 
Example : the default role for developers is bound to the business group gcp-developer_shipping-dev and applies exclusively to the "shipping-dev" project.
These default roles are automatically assigned during project creation.

- Project Specific Role for (Orga or Business) Profiles : Roles (defined at the project level) provide additional permissions not covered by the default role. They are used to meet specific project requirements that fall outside the scope of the default role.
